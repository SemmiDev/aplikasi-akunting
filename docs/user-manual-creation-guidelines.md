## Content Guidelines

### Section Structure

All sections will follow this structure:

1. **Concept explanation** - Universal, can be exercised manually (pen & paper) or Excel spreadsheet
2. **Step-by-step instruction** - Detailed with screenshots to execute the concept in the app
3. **Expected result** - What the user expects to see, with menu direction and screenshot

### Screenshot Requirements

- All screenshots must reflect the case study being explained (description and amount must correspond)
- Screenshots are generated by functional tests as side-effect of test execution
- Screenshot filename convention: `{section}-{subsection}-{description}.png`

### Screenshot Architecture

**Single-source screenshot strategy:**

**Functional Tests Generate Screenshots**
- Location: `src/test/java/.../functional/{service,seller,coffee}/`
- Profile: `@ActiveProfiles("functional")`
- Data: TestDataInitializer loads industry-seed/{industry}/seed-data/
- Output: `target/screenshots/{industry}/` (NOT committed to git)
- Purpose:
  1. Primary: Verify application behavior with real transactions
  2. Secondary: Generate screenshots for user manual
- Examples:
  - ServiceTransactionExecutionTest captures transaction forms & results
  - SellerInventoryTest captures inventory transaction screens
  - ServiceReportsTest captures financial reports
  - CoffeeProductionTest captures BOM and production orders

**Why target/ not docs/?**
- target/ is already gitignored - screenshots never committed
- CI generates fresh screenshots on every build
- Reduces repository size (no binary files)
- Local development: screenshots available in target/ for debugging
- Manual generation: UserManualGenerator reads from target/screenshots/

**Data Consistency Principle:**
- Functional tests verify application behavior AND capture screenshots
- Same test execution produces both verification and documentation
- This ensures: Screenshots in manual show data that has been functionally verified
- No separate screenshot test needed - functional tests serve dual purpose

**CI/CD Integration:**
```yaml
- name: Run functional tests (generates screenshots)
  run: ./mvnw test

- name: Generate user manual (uses target/screenshots/)
  run: ./mvnw exec:java -Dexec.mainClass="...UserManualGenerator"

- name: Deploy to GitHub Pages
  uses: actions/upload-pages-artifact@v3
  with:
    path: target/user-manual
```

**Local Development:**
```bash
# Run tests (generates screenshots to target/screenshots/)
./mvnw test

# Generate manual (reads from target/screenshots/)
./mvnw exec:java -Dexec.mainClass="com.artivisi.accountingfinance.manual.UserManualGenerator"

# View manual
open target/user-manual/index.html
```

### Language

- All content in Bahasa Indonesia
- Technical terms may use English with Indonesian explanation in parentheses
- Use consistent terminology throughout (refer to Glosarium)
